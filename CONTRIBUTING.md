# Contributing

This project is small on purpose. Contributions are welcome when they keep the repository readable, procedural, easy to run, and interesting as a compact AI-to-form experiment.

## What Fits Best

- new showcase blueprints that demonstrate a clear technique or visual idea;
- small improvements to the GUI or CLI that reduce friction;
- focused engine improvements with tests;
- documentation that helps new users build their first `.vox` model or PNG asset.

For onboarding material, see [docs/first_blueprint.md](docs/first_blueprint.md), [docs/pixel_blueprints.md](docs/pixel_blueprints.md), and [docs/magica_vox_workflow.md](docs/magica_vox_workflow.md).

## Before You Change Code

1. Read `README.md`.
2. Check the active roadmap in `todo.md`.
3. Prefer small, isolated changes over broad rewrites.
4. Keep public behavior stable unless the new contract is documented and tested.

## Before You Open A PR Or Patch

Use this repository like a small procedural tool, not like a framework rewrite target.

- confirm that the change fits the current scope: procedural voxel generation and procedural pixel art, no broader illustration-system work;
- prefer one focused improvement per change;
- if you touch behavior visible from the GUI, CLI, or the public `vox_art.engine` / `pixel_art.engine` facades, update docs and tests in the same change;
- if the change mostly adds code but does not improve the user workflow, demo set, or runtime contract, reconsider whether it belongs here.

## Blueprint Contribution Rules

- A blueprint must expose `make_model(x, y, z, W, D, H)`.
- A pixel blueprint must expose `make_image(x, y, W, H)`.
- Seed-aware blueprints may extend this to `make_model(x, y, z, W, D, H, seed=None)`.
- Seed-aware pixel blueprints may extend this to `make_image(x, y, W, H, seed=None)`.
- Optional defaults should be defined as `DEFAULT_W`, `DEFAULT_D`, and `DEFAULT_H`.
- Pixel blueprints should define `DEFAULT_W` and `DEFAULT_H`.
- Optional metadata should be defined as `BLUEPRINT_DISPLAY_NAME`, `BLUEPRINT_DESCRIPTION`, and optional `RECOMMENDED_W`, `RECOMMENDED_D`, `RECOMMENDED_H` when the recommended size differs from the defaults.
- Pixel blueprints may use `RECOMMENDED_W` and `RECOMMENDED_H` when the recommended size differs from the defaults.
- Blueprints may optionally define `BLUEPRINT_PALETTE_OVERRIDES` to embed intended colors directly into exported `.vox` files.
- Pixel blueprints may also define `BLUEPRINT_PALETTE_OVERRIDES` to control PNG palette output directly.
- Blueprint logic should be deterministic for the same inputs unless randomness is an intentional feature and is explicitly documented.
- If a blueprint supports seeded variants, different seed values should produce deterministic but distinct outputs.
- Prefer readable geometry composition over clever but opaque math.
- Avoid extremely slow per-voxel or per-pixel logic unless the visual result justifies it.

Use [templates/vox_blueprint_template.py](templates/vox_blueprint_template.py) for voxel blueprints and [templates/pixel_blueprint_template.py](templates/pixel_blueprint_template.py) for pixel blueprints.

For pixel-art contributions specifically:

- prefer transparent backgrounds unless the asset category clearly wants a tile fill;
- keep silhouettes readable at native resolution first;
- use enlarged nearest-neighbor export only as a viewing aid, not as a substitute for clean pixel decisions;
- prefer small palette counts and stable seeded variation over noisy detail.

## Engine And Tooling Changes

- Keep `save_as_vox(...)` compatible with the GUI and CLI unless the interface is intentionally redesigned.
- Treat `save_outputs(...)` as the main shared workflow entry point for CLI and automation changes.
- If you change contracts in `vox_art.engine`, update tests and documentation in the same change.
- If you change contracts in `pixel_art.engine`, update tests and documentation in the same change.
- Keep CLI `--quiet` behavior and API-level `progress=False` behavior aligned when changing logging or export flow.
- Do not introduce new dependencies without a clear reason.
- Keep pixel-art behavior inside `pixel_art/` and `pixel_engine/` rather than folding 2D logic into the voxel runtime as a hidden special case.

## Documentation And Media Changes

- Keep README claims aligned with the actual demo set and current outputs.
- If you replace showcase images in `vox_preview_assets/`, keep the same canonical filenames when possible so README links stay stable.
- Treat autogenerated preview PNGs and final repository-facing screenshots as different asset types.
- When adding screenshots, prefer images that show a distinct capability rather than more versions of the same shape family.

For media planning details, see [vox_preview_assets/README.md](vox_preview_assets/README.md) and [vox_preview_assets/SCREENSHOT_SPEC.md](vox_preview_assets/SCREENSHOT_SPEC.md).

## Current Packaging Strategy

The repository is intentionally kept as a script-first project instead of an installable package, even though the public voxel and pixel workflows now live under the packaged `vox_art/` and `pixel_art/` folders.

- prefer improving the existing scripts over introducing package boilerplate;
- avoid refactors that only make sense for `pip` distribution unless packaging becomes an explicit goal;
- if you propose a package layout later, tie it to a concrete use case such as external imports or distribution.

## Running Checks

Run the minimal test suite before proposing a change:

```bash
python -m unittest discover -s tests -v
```

On Linux or macOS:

```bash
python3 -m unittest discover -s tests -v
```

If your change affects the showcase output, also do a quick manual smoke pass on at least one representative blueprint through the intended workflow:

- GUI for launcher and validation changes;
- CLI for build/export changes;
- preview output when touching rendering or asset-facing docs.

## Demo Set Policy

The repository keeps only a small curated set of example blueprints. New examples should be added only when they show a distinct capability that is not already represented by the current demo set.

Technique-oriented examples are also welcome when they teach one clear procedural idea, such as primitive layering, negative-space carving, silhouette control, or seeded variation.

## Good First Issues

These are intentionally small tasks that fit the repository well:

- improve README screenshot captions or asset layout when the showcase set changes;
- add a small technique-oriented blueprint that demonstrates one focused idea clearly;
- improve error wording in the GUI without changing the workflow;
- add a narrow regression test for an already documented behavior;
- refine contributor documentation when a confusing step is discovered.

## Release-Oriented Contributions

Useful late-stage release contributions include:

- tightening README wording when project boundaries drift;
- polishing the showcase asset set without inflating it;
- checking that changelog, roadmap, and README do not contradict each other;
- improving contributor onboarding without turning the project into a process-heavy repository.

## Style Expectations

- keep code straightforward;
- preserve the current lightweight architecture;
- prefer explicit behavior over hidden magic;
- add tests for behavior changes, not for internal trivia.
